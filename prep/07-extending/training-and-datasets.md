# Training on a Large Dataset — the honest, complete answer

> "Can we train it on a very large dataset so it gives better responses?"

Short answer: **yes — and the pipeline is already built** (`fie generate-dataset`
→ `fie train` → `--engine ml`). But "train" means different things for the three
backends, and being precise about that is itself an interview signal. Here's the
full picture.

## First, dissolve the misconception

The system has **three interchangeable reasoning backends**, and only one of them
is "trained" in the ML sense:

| Backend | Is it "trained"? | How you make it better |
|---|---|---|
| **Rule engine** (`agent/engine.py`) | No — hand-written heuristics | Edit/version the rules; prove it with replay |
| **LLM** (Grok/Claude, `agent/llm.py`) | Not by us — the provider trained it | Better **prompt**, better **evidence summary**, few-shot examples, (optionally) fine-tuning |
| **ML classifier** (`ml/`, `agent/ml_engine.py`) | **Yes** — we train it | Feed it a bigger/better labeled dataset |

So "train on a large dataset for better responses" most literally applies to the
**ML classifier**, and the repo does exactly that. For the **LLM**, you don't
retrain the model — you improve what you feed it (and can fine-tune if justified).

## What the project actually trains (and how)

```bash
fie generate-dataset --n-per-class 500   # 8 classes -> 4,000 labeled samples
fie train --n-per-class 500              # generate + train + hold-out score
fie eval --engine ml                     # score it on the golden set (same harness)
```

- **Dataset** (`ml/dataset.py`): balanced, class-by-class, generated by jittering
  the 8 scenario archetypes (`simulator/generate.py:build_variant` — perturbs
  effect magnitudes, durations, timing, start times, and noise). You can crank
  `--n-per-class` to 5,000+ for a genuinely large set.
- **Features** (`agent/features.py`): the **same** extractor is used at train and
  serve time. This is the most important line in the ML story — it's how you
  avoid **train/serve skew**, the classic silent ML bug. The model artifact even
  stores the feature-name list and refuses to serve if it doesn't match the code.
- **Model** (`ml/train.py`): `StandardScaler → RandomForest`, held-out scored,
  saved to `data/models/`.
- **Serving** (`agent/ml_engine.py`): predicts the category + probability
  (→ confidence), then reuses the shared timeline/grounding scaffolding so its
  output is a drop-in for eval and replay.

## The honest caveat you should say out loud

On this **synthetic, cleanly-separable** data, the classifier scores ~100% — same
as the rule engine. So a bigger dataset does **not** beat the rule engine *here*.
Say that plainly, then make the real point:

> "The value isn't the accuracy number on synthetic data — it's that the ML
> engine plugs into the *same* evaluation and replay harness as the rule and LLM
> engines, with train/serve parity guaranteed by a shared feature extractor. On
> real, noisy, ambiguous plant data — where hand-written thresholds get brittle —
> that's exactly where a trained model earns its place, and I can prove any
> version change is safe before shipping it."

That answer shows you understand *when* ML helps (messy, high-dimensional,
ambiguous signals) versus when rules are better (few, well-understood signals) —
which is more valuable than "more data = better."

## The four real ways to "make it better with data"

1. **Grow the ML training set** — more variants, more noise, more asset types.
   `fie generate-dataset --n-per-class N`. Also: replace synthetic with *real*
   labeled incidents as you collect them (the pipeline is identical).
2. **Grow the evaluation/golden set** — arguably higher-value than training data.
   More labeled incidents in `data/golden/` = more confidence that any engine
   (rule/LLM/ML) is correct. Evaluation is how you *trust* the system.
3. **Improve the LLM path without retraining the LLM** — better system prompt,
   richer evidence summary (`agent/llm.py:_summarize`), and **few-shot examples**
   drawn from the golden set. This is "in-context learning": you're not changing
   weights, you're giving the model better inputs. Cheapest, fastest lever.
4. **Fine-tune an LLM** (only if justified) — take the generated/real dataset,
   format it as instruction/response pairs, and fine-tune a small open model or a
   provider's fine-tune endpoint. Costs money and MLOps effort; usually **not**
   worth it until prompt+few-shot are exhausted and you have thousands of real,
   high-quality labeled examples. Grok's free tier is for inference, not custom
   fine-tuning — so for this project, fine-tuning is a "later, if ever" option,
   not the free path.

## About the free Grok key specifically

- The free/console key gives you **inference** on xAI models via the
  OpenAI-compatible API — that's what `agent/llm.py:GrokEngine` uses.
- It does **not** train or fine-tune a model for you. "Better responses" from
  Grok come from a better prompt + evidence summary + few-shot examples, not
  from training.
- Because every LLM call falls back to the rule engine on error, a rate-limited
  or exhausted free key never breaks the demo — worth stating as a design choice.

## Training on a real Kaggle dataset (wired: AI4I 2020)

The training pipeline is no longer limited to synthetic data. There's a
real-dataset track for the **AI4I 2020 Predictive Maintenance** benchmark
(milling machine, 10k rows, five failure modes):

```bash
# download ai4i2020.csv from Kaggle/UCI (needs your login), then:
fie train --source ai4i --csv path/to/ai4i2020.csv
# mode-identification only (drop the ~96% no-failure rows):
fie train --source ai4i --csv path/to/ai4i2020.csv --failures-only
```

- Loader: `fie/ml/external.py:load_ai4i` — maps the CSV to a feature vector
  (type, air/process temp, temp difference, rpm, torque, mechanical power, tool
  wear, overstrain proxy) and the five modes → readable labels.
- It prints a **per-class precision/recall/F1 report**, not just accuracy — which
  honestly exposes the severe class imbalance and the *unlearnable* `RNF` (random
  failure) class (near-zero recall, by design — it has no signature). Being able
  to point at that and explain it is worth more than a single accuracy number.
- **Separate track by design.** AI4I lives in a different feature/label space
  than your six-signal CNC bundles, so its model is saved as `ext-ai4i-*.joblib`
  and is deliberately **not** loaded by the reconstruction MLEngine (which globs
  `ml-*`). This avoids faking a mapping that would be train/serve skew in
  disguise. The point it proves: *the training pipeline generalizes to a real
  dataset.*

### The full-integration option: Microsoft Azure PdM
[kaggle.com/datasets/arnabbiswas1/microsoft-azure-predictive-maintenance](https://www.kaggle.com/datasets/arnabbiswas1/microsoft-azure-predictive-maintenance)
is the dataset that actually matches this project's *shape* — telemetry +
error logs + maintenance + failure labels per machine over time. Wiring it means
mapping its five CSVs onto the canonical models and building windows around
failures, so it feeds the **real ingestion + reconstruction pipeline** (not just
a standalone classifier). Bigger job; it's the natural next step if you want the
end-to-end real-data story.

## If an interviewer pushes: "so is this real ML or not?"

> "The shipped default is deterministic rules, because with six well-understood
> signals that's the more honest, debuggable choice. But I built the full ML path
> — dataset generation, feature extraction with train/serve parity, training,
> and a served classifier scored by the same harness — precisely so that when the
> signal space gets big and messy, swapping in a learned model is a one-flag
> change I can validate with replay. I'd rather show I know *when* to reach for ML
> than bolt it on to look impressive."
